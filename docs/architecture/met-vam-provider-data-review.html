Met and V&A Provider Data Review

Compact audit of what each provider API offers, what Anacronia imports into local Museum Object and Image Asset records, what reaches the object detail overlay, and what is currently preserved only in raw JSON or left unused.

Review date
June 8, 2026

Scope
Online Provider Search for Met and V&A. Local folder import is out of scope.
Met is image-rights strict

We search with hasImages=true, fetch full object records, require isPublicDomain === true, then import up to three full JPEG image URLs.

V&A is metadata-richer but flatter in UI

V&A records expose nested production, place, content, warning, location, and image metadata. The current detail view shows only part of that richness.

Imported records are the source canon

Everything that becomes imported Collection/User Library material should keep its complete provider record on disk, not only the mapped fields.

Main design gap

The detail overlay has useful essentials, but the chosen priority set underuses place, production context, structured maker roles, warnings, and descriptor provenance.

Imported Material Source Canon

The raw provider record is the auditable source canon for imported online material. Anacronia can make opinionated choices for grid cards, detail overlays, descriptors, rights display, and derivatives, but those choices must not be the only copy of what the museum gave us for objects that become imported Collection/User Library material. The target rule is import-canon: if a Provider object contributes usable local Image Assets, write the complete provider object record locally and treat mapped fields as derived data.

Current status: Met and V&A records that import into local material are preserved as raw JSON. Fetched candidates that fail eligibility checks, image processing, or other import requirements can remain skipped candidates rather than becoming part of this source canon. The practical test is whether the object actually adds usable material to a Collection.

Provider field mapping and display audit
Layer Question Met current behavior V&A current behavior What detail view shows What is left out or weak Priority read
Candidate search How do we find candidate object IDs? Met /public/collection/v1/search with q={term} and hasImages=true. Only the returned objectIDs list is used. V&A /v2/objects/search with q={term}, images_exist=1, page, and page size. Only systemNumber is kept from the search result. The user sees the Provider Source, result counts, matched terms after import, and local objects. Search-page ranking details are not shown. Gap Provider search filters are not exposed or persisted as a rich query model. Met filters such as department, title-only, tags, artist/culture, medium, geolocation, date range, and on-view are unused. V&A filter richness is also unused. Good MVP simplification, but weak for controlled research sets.
Full record fetch What object payload do we fetch? Fetches the full Met object JSON from /objects/{objectID}. Live sampled records expose about sixty top-level fields including identity, public-domain flag, images, department, maker, date, medium, measurements, geography, classification, tags, links, and metadata date. Fetches V&A /v2/object/{systemNumber}, storing a top-level record plus meta. Live sampled records expose nested people, organisations, places, dates, materials, techniques, content, warnings, locations, dimensions, labels, object history, image refs, and links. Detail view receives a small normalized object contract plus image, match, and skipped-image arrays. Raw only The full record is not browsable in-app. Most provider-specific structure is available only by opening the raw JSON file on disk. Raw preservation is strong. In-app inspection is too narrow.
Raw JSON storage Where is source truth retained? Writes one raw record per object under data/met/raw-api/objects/{range}/{objectID}.json. Numeric range folders keep Met directories readable. Writes one raw record per object under data/vam/raw-api/objects/{encoded-systemNumber}.json. The provider-generic path handles string object IDs. Detail loaders read raw JSON back to populate extra fields such as accession, medium, dimensions, repository, and tags. The UI does not provide an "open raw record" affordance, and descriptors are not displayed even though they are extracted. Keep. Add a raw-record/debug link in a future developer view.
Museum Object core Which normalized object fields are stored? provider, object_id, title, object_name, artist_display_name, object_url, is_public_domain=1, rights_and_reproduction, metadata_date, raw_record_path. Same table shape. Title comes from first record.titles[].title, then _primaryTitle, then object type or ID. Artist joins person, people, and organisation maker names. Public domain is stored as false because V&A is not checked through a Met-style gate. Header title, provider label, object ID, object name, artist, provider URL, rights, metadata date, favorite state. Gap One normalized object shape hides provider-specific meaning: V&A maker roles and Met constituent roles are flattened to a single artist display string. Good for grid. Too shallow for serious object review.
Object facts Which extra object fields appear in the overlay? From raw Met JSON: artistDisplayBio, artistNationality, department, objectDate, medium, dimensions, classification, creditLine, accessionNumber, repository, and tags.term. From raw V&A JSON: production date, materials summary or materials, dimensions, categories as classification, credit line, accession number, hard-coded repository, and tags from categories/materials/techniques/styles. Artist bio, nationality, and department usually remain empty because no V&A mapping fills them. Object facts card plus expanded Provider metadata sections: Identity, Catalog description, Rights and record state, Provider tags, Provider image references. V&A briefDescription, summaryDescription, placesOfOrigin, objectHistory, physicalDescription, galleryLabels, and galleryLocations are not shown. Met geography and timeline fields are mostly not shown. Essentials are covered. Interpretive context is underweighted.
Image selection Which provider image URLs become Image Assets? Selects primaryImage first, then unique additionalImages. Imports up to DEFAULT_MAX_IMAGES_PER_OBJECT, currently three. primaryImageSmall is stored as metadata only for the primary image. Builds asset refs from record.images, then meta.images._images_meta[].assetRef, then the primary IIIF image URL tail. The first ref is primary; later refs are additional. Imports up to three. Standard image carousel, thumbnail preblur, image count control, active image card, source image link, provider image references, skipped image references. Gap Per-image V&A copyright is aggregated to object rights, not retained per image in source_metadata_json. Met image URL provenance is stored, but no source filename or image service metadata is captured beyond dimensions and URLs. Works for local analysis. Per-image provenance should improve.
Image derivatives What local image files are generated? Downloads the source JPEG, creates permanent standard-1024 and thumb-256 derivatives, validates both, captures original width and height, then discards the temporary original. Same derivative threshold using IIIF full JPEG URLs such as https://framemark.vam.ac.uk/collections/{assetRef}/full/full/0/default.jpg. Detail view uses standard-1024 for inspection and thumb-256 for loading/thumbnail support. Source originals and IIIF info.json dimensions, supported sizes, tiles, and service profile are not retained. V&A imageResolution is not stored. Correct MVP tradeoff. IIIF metadata would help image QA.
Skipped images Do we explain related provider images that are not imported? Stores skipped Met image references for beyond_max_images_per_object, collection_image_excluded, or image_processing_failed. Same skipped-reference table and reasons for V&A generated IIIF source URLs. Rights card says how many related provider images were not imported. Expanded provider image references list skipped URLs and reasons. Skipped references are not collection-scoped in the detail query, so an object imported across collections can show skipped-image history beyond the current Collection context. Useful and visible. Scope semantics need care later.
Rights How are rights and reuse signals handled? Met requires isPublicDomain === true before import. It stores rightsAndReproduction, often empty on public-domain records, and shows "Public domain" if no rights statement is present. V&A has no public-domain gate in this MVP path. It aggregates non-empty copyright strings from image metadata into the object rights statement; otherwise it uses a private-local-testing notice. Rights card and Provider metadata "Rights and record state". V&A public-domain status appears as "Not checked". Gap V&A source terms are shown as general text, not as a structured policy object. Per-image copyright is not shown beside each image. Met CC0/open-access context is not explicitly explained in detail. Meets current ADRs. Needs more precise user-facing language.
Matches How do we explain why an object appeared? Verifies search terms against title, objectName, tags, medium, culture, period, classification, and artistDisplayName. Verifies search terms against titles, object type, maker person, maker people, maker organisations, materials, techniques, categories, styles, summaryDescription, and briefDescription. Expanded "Why this object appeared" panel shows each term, verified/unverified status, and matched field names. Match field names are raw provider field paths, not friendly labels. V&A place/date/content fields are not used for match verification even when the provider search may have matched them. High-value feature. Needs human-readable field labels.
Descriptors What data supports local search across providers? Extracts descriptors from title, objectName, tags.term, medium, culture, period, classification, artistDisplayName, department, objectDate, country, region, and city. Extracts descriptors from titles, objectType, artistMakerPerson, artistMakerPeople, materials, techniques, categories, styles, placesOfOrigin, and productionDates. Descriptors are used by local result filtering. They are not listed in the detail overlay. Gap V&A maker organisations are used for match verification but omitted from descriptors. V&A content concepts/people/places, associated entities, summary/brief description, and object history are omitted. Met dynasty/reign/excavation/geographyType are omitted. Important technical layer. Should be visible for audit.
People and makers How much structured maker data is retained? Raw Met record includes display fields plus constituents with role, name, ULAN URL, Wikidata URL, and gender when available. The normalized table stores only artistDisplayName. Raw V&A record separates maker persons, groups, and organisations, each with association and notes. The normalized table joins names into one display string. Artist string, artist bio/nationality for Met, mostly empty equivalents for V&A. Role, attribution, association, authority IDs, notes, multiple maker ordering, and organisation-vs-person distinctions are not visible. Flattening is okay for grid, poor for research inspection.
Place and production How do origin and chronology differ? Met exposes geographic scalar fields such as geographyType, city, state, county, country, region, subregion, locale, locus, river, excavation, plus period, dynasty, reign, objectDate, objectBeginDate, and objectEndDate. V&A exposes nested placesOfOrigin with association and note, associated/content places, productionDates with earliest/latest, production and productionType, plus accession year. Detail mainly shows Date. V&A place is currently not shown as a fact. Some Met geography can participate in descriptor search but is not displayed. Gap This is one of the largest information-design misses. Place and production context are central to visual research but mostly absent from the object detail panel. High-priority improvement.
Descriptions and interpretation Do we show provider prose? Met sampled API records are more field-based; the current mapping shows facts and tags but no long description field. V&A often has briefDescription, summaryDescription, physicalDescription, objectHistory, historicalContext, and galleryLabels. None of these V&A prose fields appear as a dedicated description block. Gap V&A loses a lot of interpretive value in the current detail priority set. Even a compact description excerpt would improve review quality. High-priority for V&A.
Warnings and sensitivity What sensitive-content data survives? No equivalent first-class Met warning mapping in the current importer. V&A search records expose warning types and image warning summaries. Full records expose content warnings. Image metadata can include sensitiveImage. Only per-image sensitive_image is stored in source_metadata_json and shown on the Active image card when present. Object-level content warnings and search-result warning summaries are not stored as normalized metadata and not shown. Medium-high. Important for future responsible datasets.
Grid summary What is visible before opening detail? Provider, object ID route, title, object name, artist display name, image count, cover image asset, cover dimensions, favorite state. Same normalized grid summary, using V&A title/objectType/maker mapping. Object-first grid tile with thumbnail cover, title/object signals, and sibling-image indicator. Provider-specific confidence signals such as V&A image resolution, warnings, location, or Met public-domain certainty are not present in the grid. Appropriate for dense browsing. Keep grid restrained.
Detail actions What review workflow can the user perform? Open provider record, open source image, favorite object or active image, remove from Collection, delete, navigate carousel, and navigate adjacent objects. Same actions. Action buttons are in the overlay header and cards. The image stage is object-first with carousel controls when siblings exist. No "open raw JSON", "show descriptor provenance", or "copy source identity" action. Good operational base. Add audit actions later.

Met model shape

Mostly flat object records. Images are direct JPEG URLs. Rights are object-level through isPublicDomain and open-access image availability. Geographic and maker fields are mostly scalar display fields plus optional structured constituents.

V&A model shape

Nested object record plus metadata wrapper. Images are asset identifiers and IIIF services. Makers, places, dates, descriptions, content, warnings, and dimensions are often repeating structured arrays with associations and notes.

Anacronia model shape

Deliberately small shared tables: museum_objects, image_assets, object_matches, descriptors, skipped_candidates, and skipped_image_references. This keeps the app consistent but compresses provider meaning.

Recommended information-design priorities
Priority Change Why it matters Likely implementation path
High Add a visible Place and production section. Met and V&A expose this differently, but it is central to comparing objects across providers and collections. Extend provider raw field mapping in collection_objects.py. Keep normalized fields compact: production date, place made, period/culture, and source field provenance.
High Show V&A description prose when present. Current V&A detail loses useful briefDescription and related context that explains what the image/object is. Add a Provider description card using V&A briefDescription, then summaryDescription. Keep it collapsed or visually compact for dense review.
Medium Retain per-image V&A copyright in image metadata. Current object-level aggregation can hide image-level differences when an object has multiple images. Store copyright_text in image_assets.source_metadata_json and show it on the Active image card.
Medium Display descriptor provenance in detail. Descriptors drive local search, but the user cannot see what Anacronia extracted or which provider fields produced it. Add descriptors to the detail API and render a compact grouped list by descriptor type/source field.
Medium Fix V&A descriptor asymmetry for maker organisations. V&A organisation makers can verify matches but are not stored as descriptors, so local search is weaker than match explanation. Add record.artistMakerOrganisations.name.text to extract_vam_descriptors.
Low Add a developer/audit link to raw JSON. The raw record is already local and valuable when checking mapping decisions. Add a backend endpoint gated to local file paths or expose a safe "raw record path" copy affordance in development mode.
